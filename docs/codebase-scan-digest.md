# Codebase Scan Digest — Onboarding/Customization Initiative

*Five granular read-only scans, July 2026. Compressed to the facts that drive the redesign. File:line refs verified at scan time.*

## 1. Onboarding & boot
- Boot: `App.jsx` setupState machine (booting → firstrun/unlock/recovery/recover-orphan/null). First-run detection is **IDB-peek-based** (`peekStoredData` + `scanForOrphanedData`) — never localStorage; high-risk to touch. Multi-system slots via `initSystemsRegistry`.
- `StorageModeSetup` **no longer offers a mode choice or encryption** — auto-selects local, one "Enter" button, plain DB (encryption opt-in later in Settings). Contains privacy notice, importers (SP/Octocon/PluralSpace/OpenPlural/Ampersand), DataRescuePanel. Has a dead inline UnlockScreen.
- Real first-run chain: StorageModeSetup (App boot) → then **on Dashboard mount**: DisclaimerModal (`disclaimer_acknowledged_v1`) → TermsSetupModal (`terms_setup_done`) → TourModal (`tour_seen`). Split across two lifecycles.
- **Two tour systems**: TourModal ("Guide", 17 static slides) + FeatureTour ("Tour", ~108 spotlight steps, 16 sections, ~70 `data-tour` anchors; creates/deletes a temp "Tour Demo" alter; only full run sets `symphony_dailytask_tour_completed_v1`). PageTutorialBanner runs FeatureTour scoped per route (`restrictToRoute`); replayable from Settings → About.
- `RemindersOnboarding` = the app's only existing **curated preset-picker pattern** (7 seeds, first 4 pre-checked, bulkCreate) — rendered as the Reminders empty state.
- **UnlockScreen has no biometrics/paste affordances/autocomplete.** RecoveryScreen handles 6 failure kinds, exports raw before destructive reset.
- Quirks: `disclaimer_acknowledged_v1` survives Delete-All wipe (inconsistent); shared `EmptyState.jsx` exists but is unused (every page hand-rolls); TwaToNativeMigrationModal hard-disabled; AccessibilityFab mounted at z-110 on earliest screens; **no way to re-run TermsSetup/StorageModeSetup** short of full wipe.
- Settings: live appearance = `AdvancedAppearanceNew.jsx` (imported AS `AdvancedAppearance`); **dead**: `AdvancedAppearance.jsx`, `ThemeColorSettings.jsx`, `DiaryCardPresetsManager.jsx`. `ManageCheckIn` (/manage-checkin; Sections/Symptoms/Habits/Diary tabs) is buried behind Settings → Tracking setup → "Open". `TermsSettings` is much richer than TermsSetupModal (signposts + **explicit word-form overrides** — the i18n mechanism already exists). "Infer presence" toggle lives in Tracking setup.

## 2. Check-in ecosystem
- `QuickCheckInModal` (1592 lines): 8 user-configurable sections (`SystemSettings.quick_checkin_sections`, lib `quickCheckinSections.js`): feeling, fronting, activity, symptoms, diary, note, contacts, location. Collapsible-set model, prev/next nav, draft autosave (`useFormDraft`), no outside-dismiss, Android ghost-click guard. Save order: Activities → fronting sync (trigger metadata on session) → EmotionCheckIn (note >50 words → JournalEntry spin-out) → SymptomCheckIns (real `check_in_id` FK) → DiaryCard → Location → contact encounters. Edit mode reconciles symptoms; adds (not edits) activity/diary/location.
- Slider: localStorage-chosen tracked rating symptom (default Energy), deduped against symptoms section at save.
- Symptom model: `Symptom` {label, category symptom|habit, type rating|boolean, is_positive, color, order, is_default, is_archived} + `SymptomCheckIn` {symptom_id, timestamp, severity 0–5|null, check_in_id} + `SymptomSession` {start/end, is_active, severity_snapshots[], notes}. Sessions and check-ins loosely coupled (shared symptom_id only). Severity menu writes BOTH snapshot and standalone check-in.
- `SystemCheckIn` = separate "System Meeting" 5-step ritual that forks emotions into EmotionCheckIn (incl. per-participant alter_id rows!) + merges fronters into FrontingSession.
- Emotions: `WHEEL` hardcoded in EmotionWheelPicker (good/bad/neutral/body; 2 modes wheel/classic + always-on search). **Emotions are plain strings** — color/distress re-derived by label; rename = lossy. Distress = localStorage set (`symphony_system_distress`, NOT backed up, per-device) OR `CustomEmotion.is_distressing`. Grounding **never auto-launches**; post-save dialog offers navigate("/grounding"); selected emotion not carried in.
- Diary: **three config systems** — `DiaryTemplate.groups` (LIVE: DiarySection in QCM + DiaryTemplateManager), `SystemSettings.diary_sections_schema` (read by DiaryCards page, never written → always defaults), `SystemSettings.diary_sections` (dead both ends).
- Dead code: EmotionCheckInModal, SymptomManager, SymptomCard (obsolete SymptomDefinition model), DiaryCardPresetsManager.
- CheckInLog links activities/locations by **±2-min timestamp proximity**, not FK. Edit via double-click/long-press (`useEditPress`).
- Reuse: EmotionWheelPicker, SymptomsSection (onCheckInsReady getter), DiarySection/RatingRow, AlterTreeSelect/FrontPickRow/useSwipeActions, ActivityPillSelector, ContactMultiSelectList, useFormDraft, StateCheckFlow + groundingDefaults (EMOTIONAL_STATES: dissociation, derealization, anxiety, overwhelm, crisis, switching, stuck).

## 3. Profiles & customization
- AlterProfile: 10 tabs; page-level theming via custom_fields style keys; `.os-pf` wrapper + injected scoped styles. GroupProfile **reuses the entire stack** — treat alters+groups as one system.
- Alter fields beyond schema: is_pinned, emoji+use_emoji_as_alias, preset_answers, privacy_levels[], birthday (free text) + origin_year, age (no editor), avatar_rotation_mode, **banner_url (imported from SP/OpenPlural but NEVER rendered)**, subsystems_icon, friends_visible, pk_* fields, _field_order. `alter_custom_fields` polymorphic (array vs object map).
- Bio pipeline: description HTML w/ `data-blocks` JSON wrapper. BioEditor 4 modes (Plain/Simple/Blocks/Raw via MultiModeHtmlEditor). `blocksToHTML`/`htmlToBlocks` (fresh ids every call — footgun). SimplePreview = renderer: spoilersToHtml → DOMPurify (ADD_TAGS style!) → px→rem. `scopedBioStyle.js` scopes selectors + renames keyframes, drops @import/javascript:/expression(); **does NOT block position:fixed, remote url(), z-index**. Templates: `bioTemplates.js` — 8 full pages + 18 stackable modules, categories, theme-aware, data-edit fill spans; BioTemplatePicker (modules append/stack, pages replace).
- Theming: LIVE = `ThemeContext.jsx` (16 presets × light/dark × **8 roles**; sets BOTH `--color-*` hex and `--*` HSL triplets — must set both) + `AdvancedAppearanceNew` (full hex palette editor — NO contrast validation; named user presets with granular PRESET_PARTS; fonts incl. custom upload; wave; corner; per-alter theme links). `generateDarkTheme()` heuristic derivation.
- **`contrast.js` already CORRECTS colors** — `adjustForContrast` = hue-preserving minimal lightness shift; applied only to chips/badges, never palette editors.
- **`profileStyle.js` = the per-profile theming engine** (PS keys: _bg_*, _header_*, _page_*, full `_theme_*` 8-role palette + wave); `profileThemeCss` re-themes all Tailwind utilities inside `.os-pf`. `ProfileStyleEditor.jsx` = existing typed-options UI. **Strongest Tier-2 foundation.**
- Fonts: CustomFont entity + IDB bytes; app-wide @font-face injection; per-profile via _header_font/_page_font + profileFonts.js. Images: /local-image/<id> SW-served + legacy scheme; ImageAsset rotation pools (avatar/background).
- Privacy: level-based (`privacyLevels.js`: SHARE_FIELDS 9-field allowlist; levels on SystemSettings; alter.privacy_levels ids; per-friend overrides) — NOT per-field-per-alter. InfoTab eye = display-only. useAnonymizeMode = screenshot blur.
- Gaps: no typed theme schema/validation, no seed→palette generation, no contrast warnings in editors, no banner/gallery primitives, dual var namespaces (--color-* vs _theme_*), no style size caps or reset-to-safe.

## 4. Fronting & attribution
- FrontingSession: individual model (alter_id+is_primary) + legacy group model handled everywhere via `sessionNormalizer.js` (canonical; sliceByOverlap sweep-line; STALE 48h flag). Second older normalizer in frontingUtils.js. Undeclared live fields: trigger metadata, session_emotions/symptoms/note JSON payloads.
- **Attribution:** `attribution.js attributeRecord` (explicit → fronters → author → inference → none) is canonical but **barely used** — ~15 inline re-implementations. EmotionCheckIn/DiaryCard/Activity default `fronting_alter_ids` from current fronters; JournalEntry author = primary; Chat default primary→fronter→alters[0] + signposts; Bulletin defensive live-fetch. **SymptomCheckIn NEVER gets attribution at write time** → alterFingerprint symptom branch is dead code. AlterNote.alter_id = subject, not author.
- **`inferredPresence.js` + `useAuthoredPresence`** — the passive-attribution scaffolding, DEFAULT ON (localStorage `symphony_infer_presence_from_authorship`, window 30min–6h): builds {alterId, ts} events from MentionLog `authored` rows (chat/bulletins/comments — written by `saveAuthoredLog`) + JournalEntry; `inferAlters(ts)` + `buildInferredSessions()`. Already consumed by EmotionAnalytics, SymptomAnalytics §8.4, AlterActivityMatrix.
- Design principle (Analytics.jsx:254-259, fingerprint.js): fronting TIME = real sessions only; inference only for content attribution. Keep.
- Fronting-off hard-breaks: timeline alter columns, Fronting tab, CoFronting, buildFrontingSection, SwitchLogAnalytics, preSwitchSignature, `switch_logged` daily task, friends front-sync, **chat private-channel unlock** (keys off fronter membership), AlterProfile is-fronting.
- `useAnalyticsGrouping` reads localStorage `symphony_analyticsGrouping` (CLAUDE.md's claim it reads SystemSettings is WRONG).

## 5. Friends/server & data layer
- Server fully in-repo: `api/friends/*` (13 handlers) + push + reminders; Vercel serverless + Vercel KV (Upstash). Auth = {userId, secret} equality; **secrets in GET query strings**; **wildcard CORS**; front blobs plaintext on relay; member-list shares E2E-encrypted (ECDH, TOFU + manual safety numbers).
- Base URL hardcoded ×3 (friendsApi.js:22, serverReminderSync.js:33, public/runners/friends-poll.js:39). Web = same-origin → self-deploy works free; **no federation** (one KV, one identity scope). Self-hosting needs: runtime-config host → per-identity server ({server,userId,secret}, code@server) → per-server push keys → transport hardening.
- All friend strings render as escaped text today (no HTML path); sender-side-only bio stripping; `fronter.color`/`m.color` flow into inline styles unvalidated (url() beacon risk). renderBulletinContent = hand-rolled sanitizer for LOCAL content, explicitly disclaims remote defense.
- **No CSP anywhere** (index.html, vercel.json — confirmed). Network surface: own API, api.pluralkit.me, api.apparyllis.com (+CDN), octocon CDN, **arbitrary https image hosts** (avatar caching/migration paths — constrains img-src), one fonts.googleapis.com ref. PBKDF2 100k iterations (low; OWASP ≥600k).
- localDb: schemaless Proxy; filter = strict equality; every mutation re-serializes+re-encrypts whole blob; multi-system storage keys; merge dedup content keys; SystemSettings singleton fold. ENTITY_NAMES 70+; EXPORT_CATEGORIES 25. Dead: base44/ root dir, @base44/* deps in package.json (name still "base44-app").
